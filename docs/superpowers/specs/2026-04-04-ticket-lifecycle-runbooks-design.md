# Ticket Lifecycle Management & Runbook Generation Design

## Goal

Prevent stale closed tickets from polluting agent context, enforce resolution documentation on ticket close, and automatically generate runbook documents for recurring issues — with Knowledge Nexus as a tagged downstream consumer.

## Architecture

Event-driven pipeline: ticket close enforces resolution notes → async review agent evaluates recurrence and writes a runbook → `document.created` NATS event emitted → KN integration syncs with source citation. A separate hourly pattern-detection sweep handles cross-ticket recurring patterns. Paperclip is the source of truth; data flows one direction to KN.

## Tech Stack

- PostgreSQL (Drizzle ORM) — schema changes
- Existing documents + document revisions system — runbook storage
- Existing routine run machinery — review agent execution
- NATS (nuntius) — event emission
- `pg_trgm` — trigram similarity for pattern detection (no new dependencies)
- KN integration plugin — downstream consumer (out of scope for this spec)

## 1. Data Model Changes

### 1.1 `issues` table — add `resolution_notes`

```sql
ALTER TABLE "issues" ADD COLUMN "resolution_notes" text;
```

- Nullable in DB; enforced as required by the service layer on status transitions to `done` or `cancelled`.
- Stored as plain text (markdown accepted).

### 1.2 `documents` table — add `kind` and `source_issue_id`

```sql
ALTER TABLE "documents"
  ADD COLUMN "kind" text NOT NULL DEFAULT 'general',
  ADD COLUMN "source_issue_id" uuid REFERENCES "issues"("id") ON DELETE SET NULL;

CREATE INDEX "documents_kind_idx" ON "documents" ("company_id", "kind");
CREATE INDEX "documents_source_issue_idx" ON "documents" ("source_issue_id");
```

- `kind`: `"general"` (default, existing docs) or `"runbook"` (generated by this feature).
- `source_issue_id`: the canonical originating ticket for a runbook. Nullable. For pattern-detected runbooks spanning multiple tickets, set to the earliest ticket in the group; related issues are referenced by markdown links in the document body.

### 1.3 Drizzle schema updates

- `packages/db/src/schema/issues.ts`: add `resolutionNotes: text("resolution_notes")`
- `packages/db/src/schema/documents.ts`: add `kind`, `sourceIssueId` fields with index definitions

## 2. Default Status Filter (Agent Context)

### 2.1 Agent-facing `issues.list()` tool

In `server/src/services/plugin-host-services.ts`, the `issues.list` handler gains a default status filter:

```typescript
async list(params) {
  const companyId = ensureCompanyId(params.companyId);
  await ensurePluginAvailableForCompany(companyId);
  const filters = {
    ...params,
    status: params.status ?? "backlog,todo,in_progress,in_review,blocked",
  };
  return applyWindow((await issues.list(companyId, filters as any)) as Issue[], params);
}
```

- If no `status` param is provided, only active (non-terminal) statuses are returned.
- Agents can opt into terminal statuses by passing `status: "done,cancelled"` or `status: "all"` explicitly.
- `status: "all"` is a new sentinel value handled in `issues.list()` that skips the status condition entirely.

### 2.2 Human-facing REST API

`GET /api/issues` is **unchanged**. Board users see all statuses by default. Only the agent tool changes.

## 3. Resolution Notes Enforcement

### 3.1 Service layer

In `server/src/services/issues.ts`, the `update()` function checks for resolution notes before applying a terminal status transition:

```typescript
const isClosing = (patch.status === "done" || patch.status === "cancelled");
const hasNotes = patch.resolutionNotes?.trim();
if (isClosing && !hasNotes) {
  throw badRequest("Resolution notes are required when closing or cancelling an issue.");
}
```

- Applies to both agent-facing and human-facing update paths (single enforcement point in the service).
- Returns HTTP 400 with a clear message.

### 3.2 Cascade/bulk operations exception

System-initiated bulk cancellations (e.g., project deletion cascade, agent termination cleanup) pass a system-generated resolution note:

```typescript
resolutionNotes: "Cancelled: parent project deleted."
```

This prevents enforcement from breaking internal cascade operations.

## 4. Review Agent Trigger

### 4.1 Trigger point

After a successful terminal status transition with valid resolution notes, the issue service asynchronously enqueues a routine run — no `await`, never blocks the close response:

```typescript
void enqueueRunbookReview(companyId, issueId, contextSnapshot);
```

### 4.2 Context snapshot

The snapshot is built once at close time and passed as the routine trigger payload:

```typescript
{
  issueId: string;
  identifier: string;          // e.g. "ENG-123"
  title: string;
  description: string | null;
  resolutionNotes: string;
  status: "done" | "cancelled";
  projectId: string;
  labelIds: string[];
  comments: Array<{ authorAgentId, authorUserId, body, createdAt }>;
  assigneeAgentId: string | null;
  issueUrl: string;            // canonical Paperclip URL
  closedAt: string;            // ISO timestamp
}
```

No DB queries needed by the review agent — the full context is in the payload.

### 4.3 Runbook Review routine

A system-seeded routine named `__system_runbook_review__` is auto-created per company on first use (not user-visible, not editable by users). It is assigned to the company's primary system agent (the first agent with `kind: "system"` for the company, falling back to any active agent). It runs the review agent with the snapshot as input.

The agent prompt instructs:
1. Evaluate whether this issue is likely to recur based on its nature, resolution, and notes.
2. If likely to recur: write a markdown runbook covering symptoms, root cause, resolution steps, and prevention. Call `documents.create()` with `kind: "runbook"` and `sourceIssueId`.
3. If unlikely to recur: exit with no artifact.

### 4.4 Async guarantee

The trigger is fire-and-forget from the close handler. If the routine fails, it follows the existing routine failure path (logged, retryable). Ticket close always succeeds independently.

## 5. Pattern Detection Sweep

### 5.1 Schedule

A system-level interval (not a user routine) runs every 60 minutes via `setInterval` at app startup, similar to the existing feedback export flush.

### 5.2 Detection logic

```
1. Fetch all issues closed in the last 7 days (status done/cancelled, updatedAt >= now - 7d)
2. Group by (projectId, labelIds) as a coarse filter
3. Within each group, compute pg_trgm similarity on (title || ' ' || resolutionNotes)
4. Clusters with similarity >= 0.4 and count >= 2 are "patterns"
5. For each pattern cluster:
   a. Check if a runbook already exists with source_issue_id = earliest issue in cluster
   b. If not: trigger runbook generation via the same review agent routine,
      passing the full cluster as context (all issue snapshots)
   c. If yes: create a new document revision appending the new case as evidence
```

### 5.3 Similarity threshold

- Trigram similarity threshold: `0.4` (configurable via env var `RUNBOOK_SIMILARITY_THRESHOLD`, default `0.4`)
- Minimum cluster size: `2` tickets
- Lookback window: `7` days (configurable via `RUNBOOK_SWEEP_LOOKBACK_DAYS`, default `7`)

## 6. Document Events & KN Integration Contract

### 6.1 Event emission

After a runbook document is created or updated, Paperclip publishes to NATS:

**Subject:** `paperclip.documents.runbook.created` or `paperclip.documents.runbook.updated`

**Payload:**
```json
{
  "companyId": "uuid",
  "documentId": "uuid",
  "documentUrl": "https://<host>/companies/<companyId>/documents/<documentId>",
  "sourceIssueId": "uuid",
  "sourceIssueIdentifier": "ENG-123",
  "kind": "runbook",
  "title": "string",
  "revisedAt": "ISO 8601 timestamp"
}
```

### 6.2 KN integration contract (out of scope — defined here as boundary)

The KN integration plugin:
1. Subscribes to `paperclip.documents.runbook.created` and `paperclip.documents.runbook.updated`
2. Fetches full document body from Paperclip documents API using `documentId`
3. Stores in KN tagged with:
   - `source: "paperclip"`
   - `paperclipDocumentId: <uuid>`
   - `paperclipDocumentUrl: <canonical URL>`
   - `paperclipCompanyId: <uuid>`
4. On `.updated`, overwrites the KN copy by `paperclipDocumentId`

**Source of truth:** Paperclip. Data flows Paperclip → KN only. KN never writes back to Paperclip.

If KN's copy diverges, re-fetch from `documentUrl`. The URL is the canonical citation.

## 7. Error Handling

| Scenario | Behavior |
|---|---|
| Agent closes ticket without resolution notes | 400 Bad Request, ticket not closed |
| Review agent routine fails | Logged as routine failure, no runbook created, ticket remains closed |
| Pattern sweep fails for one cluster | Logged, other clusters continue processing |
| NATS publish fails | Logged as warning, document still created in Paperclip (KN sync can catch up via sweep) |
| KN integration unavailable | NATS event persisted in JetStream, replayed when KN comes back |
